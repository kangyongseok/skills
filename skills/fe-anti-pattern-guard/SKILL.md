# FE Anti-Pattern Guard

프롬프트 기반 개발에서 반복적으로 발생하는 React 안티패턴을 잡기 위한 체크리스트. React 컴포넌트/훅을 작성·수정·리뷰할 때마다 아래 항목을 점검한다.

## 1. Props Drilling 지양

서버에서 가져온 데이터(캐시 가능한 서버 상태)를 여러 단계의 컴포넌트를 거쳐 props로 내려주지 않는다. 대신 데이터가 필요한 컴포넌트에서 직접 TanStack Query의 `useQuery`(또는 동등한 훅)를 호출한다.

- 대상: 서버 상태(API 응답, 캐시 가능한 데이터)
- 대상 아님: 순수 UI 상태(모달 open/closed, 폼 입력값 등) — 이런 것까지 강제로 없앨 필요는 없다.

## 2. 중첩 삼항연산자 금지

삼항연산자를 중첩해서 조건 분기를 표현하지 않는다. 조기 반환(early return), if/else, 또는 조건을 별도 함수/변수로 추출해서 가독성을 확보한다.

## 3. 파일당 단일 책임

하나의 파일에 여러 컴포넌트/훅/유틸이 뒤섞이지 않도록 한다. "이 파일이 무슨 역할을 하는지 한 문장으로 설명이 안 된다"면 분리 신호로 판단한다.

## 4. useMemo/useCallback 과사용 지양

실측으로 확인되는 리렌더링 비용이 없거나, 참조 동일성이 실제로 하위 컴포넌트 최적화(예: `React.memo`)에 영향을 주지 않는 곳에는 추가하지 않는다. "혹시 몰라서" 붙이지 않는다.

## 5. useEffect 오남용 지양

파생 상태 계산이나 이벤트 핸들러로 대체 가능한 로직을 `useEffect`에 넣지 않는다. `useEffect`는 외부 시스템과의 동기화(구독, DOM 직접 조작, 서버 통신 등) 목적일 때만 사용한다.

## 6. 콜로케이션 원칙

테스트 파일과 스토리북 파일은 `__tests__/`, `stories/` 같은 별도 폴더가 아니라 대상 코드와 같은 디렉토리에 위치시킨다. 연관성 있는 것들은 물리적으로 가까운 곳에 둔다.

## 7. Rule of Three

재사용이 2~3회 이상 실제로 발생하기 전까지는 공용(상위) 모듈로 성급하게 추출하지 않는다. 섣부른 추상화보다 약간의 중복을 허용한다.

## 8. 애매한 판단은 임의로 처리하지 않고 먼저 질문

위 1~7번 규칙 적용 여부가 애매한 경우, 주석만 남기고 넘어가지 말고 AskUserQuestion 등으로 사용자에게 먼저 확인한 뒤 진행한다.

## 9. 레거시/중복 코드는 주석으로 표시만 하고 판단은 사용자에게

작업 중 우연히 발견한 레거시 코드나 중복 코드는 스스로 삭제하거나 방치하지 않는다. `// LEGACY: 중복/미사용 가능성 있음 - 삭제 여부 검토 필요` 형태의 주석으로 표시만 하고, 삭제 여부는 사용자가 직접 판단하게 한다.

규칙 8과의 차이: 규칙 8은 현재 작업 중인 코드의 스타일 판단이 애매할 때 즉시 질문하는 것이고, 규칙 9는 작업과 무관하게 우연히 발견된 레거시/중복 코드를 주석으로 표시해 작업 흐름을 끊지 않으면서 사용자가 나중에 인지·판단하도록 하는 것이다.
